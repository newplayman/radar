# 庄家雷达 v0.4 信号追踪与回测 Implementation Plan

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task.

**Goal:** 给 v0.3 已产生的雷达信号增加自动追踪、收益归因、历史回测统计和 Telegram 每日复盘，让“庄家雷达”从异动播报升级为可验证的信号系统。

**Architecture:** v0.4 不改动 v0.3 的实时扫描主链路，只在信号落库后新增 `signal_tracking` 追踪表、价格快照采集器、追踪任务 CLI、统计聚合器和复盘 formatter。PostgreSQL 是运行主存储，SQLite 保留测试/fallback；所有外部 API 读取必须有频控、批量化、限流降级和幂等更新。

**Tech Stack:** Python stdlib、PostgreSQL 16 JSONB、SQLite fallback、Binance Futures public ticker/klines、OKX CLI/public market data、systemd timer、pytest。

---

## v0.4 Product Scope

### 必须交付

1. 每条 `RadarSignal` 保存后可生成一个 tracking record。
2. 自动追踪信号触发后：`15m`、`1h`、`4h`、`24h` 的价格表现。
3. 统计每个信号在窗口内的：
   - return_pct
   - max_runup_pct
   - max_drawdown_pct
   - hit_success，默认方向先按信号类型推断
4. 支持命令：
   - `banker-radar track-signals`
   - `banker-radar backtest-report`
   - `banker-radar telegram-review --period yesterday`
5. Telegram 每日复盘格式。
6. PostgreSQL schema + SQLite fallback 同步支持。
7. API 频控与自恢复：不为了补历史数据无限请求；限流时跳过本轮，下一轮继续。

### 暂不做 / YAGNI

1. 不做复杂策略回测撮合，不计算真实滑点/手续费。
2. 不做自动交易。
3. 不做多周期参数寻优。
4. 不接入第三方付费行情源。
5. 不要求分钟级精确最大回撤，v0.4 用低频 K 线近似即可。

## Key Design Decisions

### 1. 方向推断先简单、可解释

新增工具函数：

```python
def infer_signal_direction(kind: str, metadata: dict | None = None) -> str:
    if kind in {"空头燃料", "暗流吸筹", "链上聪明钱", "链上链下共振"}:
        return "long"
    if kind in {"空头压制"}:
        return "short"
    return "neutral"
```

v0.4 先以 `long/short/neutral` 计算表现。`综合异动` 默认 `neutral`，只统计不算胜率，除非 metadata 中已经有 direction。

### 2. 追踪窗口固定、配置化

默认窗口：

```yaml
tracking:
  enabled: true
  windows_minutes: [15, 60, 240, 1440]
  success_threshold_pct: 2.0
  max_signals_per_run: 50
  max_price_requests_per_run: 80
  price_source_order:
    - binance
    - okx
  cooldown_seconds_after_rate_limit: 900
  scheduler:
    track_interval_minutes: 15
    daily_review_time_cst: "09:05"
```

### 3. 价格采集要尽量批量化

优先用 Binance Futures `ticker/24hr` 或 `klines`。单币单窗口需要历史价格时，使用 `fetch_klines(symbol, "15m", limit=...)` 计算触发后区间的 close/high/low。OKX 用作 fallback。

### 4. 追踪是幂等任务

`track-signals` 可反复运行：

- 已完成窗口不重复更新。
- 未到期窗口跳过。
- API 失败/限流保留 pending，下次继续。
- 单条错误不影响其他信号。

### 5. v0.4 必须补齐的风控、限流和一致性设计

本节是实现前的强制 hardening checklist。v0.4 不做自动交易，但仍要避免“错误信号被过度解释”“免费 API 被打爆”“复盘数据不可信”三类问题。

#### 5.1 风控与信号解释安全

1. **复盘不等于可交易收益**：Telegram review 必须显示“未计滑点/手续费/真实成交，仅为信号后价格表现统计”。
2. **样本量门槛**：`backtest-report`/`telegram-review` 对 `total < min_samples_for_summary` 的分组标记为“样本不足”，不排序为优秀策略。
3. **极端波动标记**：如果 `abs(return_pct)`、`max_runup_pct` 或 `abs(max_drawdown_pct)` 超过 `outlier_return_pct`，保留记录但在 metadata 标记 `is_outlier=true`，摘要中同时输出“含极端样本”。
4. **方向置信度**：`infer_signal_direction` 返回的不只是 direction，还应在 metadata 中记录 `direction_source`：`metadata_override` / `kind_rule` / `neutral_default`，避免把“综合异动”误当确定方向。
5. **中性信号不计胜率**：`neutral` 只统计平均后续波动，不纳入 win_rate denominator。
6. **风险标签保留**：tracking record 必须复制原信号的 `risk`、`score`、`source` 到 metadata，后续可按高风险/低风险分组复盘。
7. **每日复盘防误导**：排序默认按窗口、样本量、平均收益展示，不能只展示表现最好的单个币；必须展示样本数、胜率、平均回撤。

#### 5.2 限流、配额和自恢复

1. **每轮总预算**：继续使用 `tracking.max_price_requests_per_run`，达到预算后停止本轮，剩余 pending 下轮继续。
2. **每 provider 预算**：新增 `tracking.max_requests_per_provider_per_run`，避免 Binance/OKX 某一个源被集中打满。
3. **K 线批量与缓存**：同一个 `symbol + provider + interval` 在同一轮 tracking cycle 内只能请求一次；多个窗口从同一批 klines 中切片计算，避免 15m/1h/4h/24h 重复请求。
4. **识别 Retry-After**：如果 HTTP 返回 429 或 header/body 包含限流信息，优先使用 `Retry-After`，否则用配置的 `cooldown_seconds_after_rate_limit`。
5. **持久化 cooldown**：systemd `Type=oneshot` 每次是新进程，内存 cooldown 会丢失；必须把 provider cooldown 写入 PostgreSQL 表或本地 JSON 文件，例如 `provider_cooldowns(provider, blocked_until, reason)`。SQLite fallback 可使用表或 `data/provider_cooldowns.json`。
6. **指数退避上限**：连续失败 provider 使用 bounded exponential backoff，最大不超过 `max_provider_cooldown_seconds`。
7. **失败分类**：区分 `rate_limited`、`network_error`、`symbol_not_found`、`no_klines`、`parse_error`。只有临时错误保持 pending；永久错误可标记 `failed_permanent`，避免无限重试。
8. **历史欠账上限**：新增 `tracking.max_tracking_age_hours`，超过该年龄仍未完成的窗口标记 `expired`，不再补请求。
9. **系统层防重叠只是第二道锁**：systemd `flock` 保留；数据库层仍要用状态流转/行锁保证并发安全。

#### 5.3 数据一致性与可重复复盘

1. **明确状态机**：`pending -> in_progress -> completed | failed_retryable | failed_permanent | expired`。不要只用 pending/completed，避免并发 worker 重复处理。
2. **PostgreSQL 并发领取**：`due_tracking_records` 应在事务中使用 `SELECT ... FOR UPDATE SKIP LOCKED`，并立即把记录改为 `in_progress`。
3. **唯一约束不够**：保留 `UNIQUE(signal_id, window_minutes)`，同时 enqueue 使用 `INSERT ... ON CONFLICT DO NOTHING`，确保重复扫描/重复 enqueue 幂等。
4. **entry_price 不能长期为 0**：如果信号本身没有 entry_price，tracking 应用信号时间附近的首根 K 线 close 回填；回填失败时标记 `failed_retryable` 或 `failed_permanent`，不能无限 skip。
5. **时间对齐**：所有 DB 时间使用 UTC `TIMESTAMPTZ`；报告层再转 CST。K 线窗口使用 `[signal_ts, due_ts]`，并记录 `observation_start_ts`、`observation_end_ts`、`provider_interval`。
6. **价格源一致性**：优先使用与原信号 source 对应的交易所；如果 fallback 到另一 provider，metadata 记录 `price_provider` 和 `fallback_reason`。
7. **symbol normalization**：统一 `BTCUSDT`、`BTC-USDT-SWAP`、链上 token symbol 的 canonical symbol，tracking 只对有合约价格源的 symbol 建任务。
8. **每日复盘幂等**：新增 `review_sends(period_start, period_end, channel, sent_at, content_hash)` 或等价机制；`telegram-review` 默认同一 period/channel 只发送一次，`--force` 才重复发送。
9. **可审计原始数据**：每条 completed tracking metadata 至少保存：`price_provider`、`kline_count`、`interval`、`observed_close_time`、`request_ts`。
10. **数据保留策略**：v0.4 可先不做清理，但计划中应预留 `tracking.retention_days`，避免长期 PostgreSQL 无限膨胀。

新增配置建议：

```yaml
tracking:
  min_samples_for_summary: 5
  outlier_return_pct: 80
  max_requests_per_provider_per_run: 50
  max_provider_cooldown_seconds: 3600
  max_tracking_age_hours: 72
  retention_days: 180
  require_review_idempotency: true
```

## Database Schema

### PostgreSQL

```sql
CREATE TABLE IF NOT EXISTS signal_tracking (
  id BIGSERIAL PRIMARY KEY,
  signal_id BIGINT REFERENCES radar_signals(id) ON DELETE CASCADE,
  symbol TEXT NOT NULL,
  kind TEXT NOT NULL,
  source TEXT NOT NULL,
  direction TEXT NOT NULL DEFAULT 'neutral',
  signal_ts TIMESTAMPTZ NOT NULL,
  entry_price DOUBLE PRECISION NOT NULL DEFAULT 0,
  window_minutes INTEGER NOT NULL,
  due_ts TIMESTAMPTZ NOT NULL,
  status TEXT NOT NULL DEFAULT 'pending',
  observed_price DOUBLE PRECISION,
  high_price DOUBLE PRECISION,
  low_price DOUBLE PRECISION,
  return_pct DOUBLE PRECISION,
  max_runup_pct DOUBLE PRECISION,
  max_drawdown_pct DOUBLE PRECISION,
  success BOOLEAN,
  error TEXT NOT NULL DEFAULT '',
  retry_count INTEGER NOT NULL DEFAULT 0,
  locked_at TIMESTAMPTZ,
  completed_at TIMESTAMPTZ,
  expired_at TIMESTAMPTZ,
  observation_start_ts TIMESTAMPTZ,
  observation_end_ts TIMESTAMPTZ,
  price_provider TEXT NOT NULL DEFAULT '',
  provider_interval TEXT NOT NULL DEFAULT '',
  metadata_json JSONB NOT NULL DEFAULT '{}'::jsonb,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE(signal_id, window_minutes),
  CHECK (status IN ('pending', 'in_progress', 'completed', 'failed_retryable', 'failed_permanent', 'expired')),
  CHECK (window_minutes > 0),
  CHECK (entry_price >= 0)
);

CREATE INDEX IF NOT EXISTS idx_signal_tracking_due
ON signal_tracking(status, due_ts);

CREATE INDEX IF NOT EXISTS idx_signal_tracking_symbol_ts
ON signal_tracking(symbol, signal_ts DESC);

CREATE INDEX IF NOT EXISTS idx_signal_tracking_kind_ts
ON signal_tracking(kind, signal_ts DESC);
```

### Provider cooldown and daily review idempotency tables

```sql
CREATE TABLE IF NOT EXISTS provider_cooldowns (
  provider TEXT PRIMARY KEY,
  blocked_until TIMESTAMPTZ NOT NULL,
  reason TEXT NOT NULL DEFAULT '',
  retry_count INTEGER NOT NULL DEFAULT 0,
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE IF NOT EXISTS review_sends (
  id BIGSERIAL PRIMARY KEY,
  period_start TIMESTAMPTZ NOT NULL,
  period_end TIMESTAMPTZ NOT NULL,
  channel TEXT NOT NULL,
  content_hash TEXT NOT NULL,
  sent_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  metadata_json JSONB NOT NULL DEFAULT '{}'::jsonb,
  UNIQUE(period_start, period_end, channel)
);
```

### SQLite fallback

同等字段，`metadata_json TEXT`，`success INTEGER`。

**Step 3: Implement**

```python
def _pct(new: float, old: float) -> float:
    return ((new - old) / old * 100) if old else 0.0


def calculate_performance(direction: str, *, entry: float, observed: float, high: float, low: float, success_threshold_pct: float) -> dict:
    if direction == "short":
        ret = _pct(entry, observed)
        runup = _pct(entry, low)
        drawdown = _pct(entry, high)
    else:
        ret = _pct(observed, entry)
        runup = _pct(high, entry)
        drawdown = _pct(low, entry)

    success = None if direction == "neutral" else ret >= success_threshold_pct
    return {
        "return_pct": ret,
        "max_runup_pct": runup,
        "max_drawdown_pct": drawdown,
        "success": success,
    }
```

## Task 13: Add production systemd tracking timer plan

**Objective:** Enable automatic v0.4 tracking cycles every 15 minutes after smoke tests pass, without overlapping runs or exhausting public API quotas.

**Existing deployment context:**

Current v0.2/v0.3 production units already use these conventions:

- `WorkingDirectory=/root/banker-radar`
- `EnvironmentFile=/root/banker-radar/.env`
- `--config /root/banker-radar/configs/radar.yaml`
- `--db /root/banker-radar/data/radar.db`
- hourly Telegram scan is handled by `banker-radar-telegram.timer`
- mention query bot is handled by `banker-radar-telegram-bot.service`

Keep the v0.4 timers aligned with those conventions.

**Files:**
- Create outside repo: `/etc/systemd/system/banker-radar-tracking.service`
- Create outside repo: `/etc/systemd/system/banker-radar-tracking.timer`
- Optional repo helper: `deploy/systemd/banker-radar-tracking.service.example`
- Optional repo helper: `deploy/systemd/banker-radar-tracking.timer.example`

**Tracking service design notes:**

1. Use `Type=oneshot` because one tracking cycle should finish and exit.
2. Use `flock` so a slow cycle cannot overlap with the next 15-minute cycle.
3. Pass explicit `--config` and `--db` to match existing units.
4. Let the application-level request budget enforce API limits:
   - `tracking.max_signals_per_run`
   - `tracking.max_price_requests_per_run`
   - provider cooldown after rate limit
5. Use modest systemd hardening that should not break root-owned project access.
6. Do not enable until:
   - `pytest -q` passes
   - `banker-radar track-signals --max-signals 5` passes manually
   - PostgreSQL/SQLite storage is verified

**Tracking service:**

```ini
[Unit]
Description=Banker Radar v0.4 signal tracking cycle
Wants=network-online.target
After=network-online.target postgresql.service
Documentation=https://github.com/newplayman/radar

[Service]
Type=oneshot
WorkingDirectory=/root/banker-radar
EnvironmentFile=/root/banker-radar/.env
ExecStart=/usr/bin/flock -n /run/banker-radar-tracking.lock /usr/local/bin/banker-radar track-signals --config /root/banker-radar/configs/radar.yaml --db /root/banker-radar/data/radar.db
TimeoutStartSec=10min
Nice=5
IOSchedulingClass=best-effort
IOSchedulingPriority=7
NoNewPrivileges=true
PrivateTmp=true
ProtectSystem=full
ReadWritePaths=/root/banker-radar /run
StandardOutput=journal
StandardError=journal
```

**Tracking timer:**

```ini
[Unit]
Description=Run Banker Radar v0.4 signal tracking every 15 minutes

[Timer]
OnBootSec=5min
OnUnitActiveSec=15min
AccuracySec=1min
RandomizedDelaySec=30s
Persistent=true
Unit=banker-radar-tracking.service

[Install]
WantedBy=timers.target
```
