# v0.3 Smart Money 与共振信号 Implementation Plan

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task.

**Goal:** 为庄家雷达接入 Binance Web3 / 链上聪明钱数据，并产出 `🧠 链上聪明钱榜` 与 `🧬 链上链下共振榜`。

**Architecture:** v0.3 不重写现有 v0.2 扫描链路，而是在现有 `MarketFeature -> RadarSignal -> formatter -> Telegram` 流程旁边新增 Smart Money 数据模型、采集器、评分器与合并器。链上信号先做“可选增强”：采集失败时不影响 OKX/Binance 合约雷达正常播报。

**Tech Stack:** Python 3.12, stdlib `urllib` + `subprocess`, dataclasses, PostgreSQL 16 on local host port 5433, pytest, existing Binance/OKX collectors, GMGN CLI fallback collectors, existing Telegram service.

---

## Scope

### In Scope

- 新增 Smart Money 数据模型。
- 新增链上 collector 抽象：优先 Binance Web3；如接口不可用/限流，降级使用本机已安装 `gmgn-cli`。
- 新增 Token Audit / GMGN token security 风险过滤模型。
- 新增 Meme Rush / 热度输入的轻量模型。
- 新增链上信号评分：`链上聪明钱`、`鲸鱼介入`、`已出货风险`。
- 新增跨源共振合并：合约信号 + 链上信号同标的合并为 `链上链下共振`。
- Formatter 新增两个榜单：`🧠 链上聪明钱榜`、`🧬 链上链下共振榜`。
- PostgreSQL 存储 signal metadata，便于 v0.4 回测使用；SQLite 仅作为测试/本地 fallback。
- CLI 支持 `--no-smart-money` 和配置开关。
- Telegram 自动推送沿用现有服务，不改 systemd 单元。

### Out of Scope

- 不做自动交易。
- 不做链上地址全量追踪。
- 不引入付费 API。
- 不做 v0.4 回测统计，只为回测预留字段。
- 不强依赖 Binance Web3：失败时先尝试 GMGN，GMGN 也失败时降级为 v0.2 行为。

## Acceptance Criteria

- `python3 -m pytest -q` 全部通过。
- `banker-radar scan` 默认使用 PostgreSQL；Smart Money 不可用时仍正常输出 v0.2 风格报告。
- `banker-radar scan --no-smart-money` 明确跳过链上模块。
- mock Smart Money 数据时，报告包含：
  - `🧠 链上聪明钱榜`
  - `🧬 链上链下共振榜`
- 同一 symbol 同时存在 OKX/Binance 合约异动与 Smart Money buy 时，最终输出只保留合并后的高分共振信号，避免重复刷屏。
- 高风险 Token Audit 标的不进入正向聪明钱榜，只能进入风险提示或被过滤。
- `.env`、数据库 dump、token 不入库；`DATABASE_URL` 只放本机 `.env` / systemd EnvironmentFile。

## Data Model Design

Add to `src/banker_radar/models.py`:

```python
@dataclass(frozen=True)
class SmartMoneyFeature:
    symbol: str
    chain: str
    token_address: str
    side: str  # buy | sell
    smart_wallet_count: int
    whale_count: int
    trigger_price: float
    current_price: float
    max_gain_pct: float
    exit_rate_pct: float
    volume_usd: float
    tags: tuple[str, ...] = ()
    source: str = "binance-web3"

@dataclass(frozen=True)
class TokenAuditResult:
    symbol: str
    chain: str
    token_address: str
    risk_level: str  # low | medium | high | unknown
    is_honeypot: bool = False
    can_sell: bool = True
    reason: str = ""

@dataclass(frozen=True)
class HeatFeature:
    symbol: str
    score: int
    reason: str
    source: str = "meme-rush"
```

Extend `RadarSignal` minimally:

```python
metadata_json: str = ""
```

Migration note: `RadarSignal` currently has positional construction in `scoring.py`; add the new field at the end with default so old calls remain compatible.

## Task 2: Add Binance Web3 Collector Interface

**Objective:** Create a provider-neutral chain collector that can parse Smart Money JSON rows from Binance Web3 and GMGN while remaining testable without live network.

**Files:**
- Create: `src/banker_radar/collectors/binance_web3.py`
- Create: `src/banker_radar/collectors/gmgn.py`
- Create: `src/banker_radar/collectors/chain.py`
- Test: `tests/test_chain_collectors_v03.py`

**Design:**

Expose pure parser functions first:

```python
def parse_smart_money_rows(rows: list[dict]) -> list[SmartMoneyFeature]: ...
def parse_token_audit_row(row: dict) -> TokenAuditResult: ...
def parse_heat_rows(rows: list[dict]) -> list[HeatFeature]: ...
```

Then expose live functions with safe failure:

```python
def fetch_smart_money(limit: int = 30) -> list[SmartMoneyFeature]: ...
def fetch_token_audit(symbol: str, chain: str, token_address: str) -> TokenAuditResult | None: ...
def fetch_meme_heat(limit: int = 30) -> list[HeatFeature]: ...

# GMGN fallback/provider functions
def fetch_gmgn_smart_money(chains: list[str], limit: int = 100) -> list[SmartMoneyFeature]: ...
def fetch_gmgn_security(chain: str, token_address: str) -> TokenAuditResult | None: ...
def fetch_gmgn_heat(chains: list[str], limit: int = 100) -> list[HeatFeature]: ...
```

**Implementation guidance:**
- Prefer Binance Web3 first, but add GMGN fallback through `subprocess.run(["gmgn-cli", ... , "--raw"])`.
- Return `[]` on collector failure and log a short warning; do not crash `scan`.
- Normalize symbols to `XXXUSDT` where possible so they can match Binance/OKX futures symbols.
- Use GMGN commands verified on host:
  - `gmgn-cli track smartmoney --chain sol|bsc|base|eth --limit 100 --side buy --raw`
  - `gmgn-cli market signal --chain sol|bsc --raw`
  - `gmgn-cli market trending --chain sol|bsc|base|eth --interval 1h --limit 100 --raw`
  - `gmgn-cli token security --chain sol|bsc|base|eth --address <addr> --raw`

**Steps:**
1. Write parser tests with sample Binance Web3 and GMGN raw rows containing smart buy, whale buy, sell, exitRate, maxGain, tags.
2. Implement pure parsers first.
3. Add live functions as thin wrappers that can be monkeypatched in tests.
4. Add provider fallback aggregator in `collectors/chain.py`: Binance Web3 -> GMGN -> empty list.
5. Run: `python3 -m pytest tests/test_chain_collectors_v03.py -q`.
6. Commit: `feat: add chain collectors with gmgn fallback`.

## Task 3: Add Smart Money Scoring

**Objective:** Convert Smart Money features into `RadarSignal` objects.

**Files:**
- Create: `src/banker_radar/signals/smart_money.py`
- Test: `tests/test_smart_money_scoring.py`

**Rules v0.3:**

1. `链上聪明钱`
   - `side == "buy"`
   - `smart_wallet_count >= min_smart_wallets`
   - `exit_rate_pct <= max_exit_rate_pct`
   - audit not high risk

2. `鲸鱼介入`
   - same as buy signal
   - `whale_count >= 1` or tags include `Whale Buy`

3. `已出货风险`
   - `side == "sell"` OR `exit_rate_pct > max_exit_rate_pct`

4. Score formula:

```text
base 45
+ smart_wallet_count * 6, max 24
+ whale_count * 10, max 20
+ fresh bonus if max_gain_pct <= max_gain_pct_for_fresh_signal, max 10
+ heat bonus, max configured
- audit penalty for medium/unknown risk
```

**Steps:**
1. Write tests for buy, whale buy, high exit risk, high audit block.
2. Implement `score_smart_money(feature, audit=None, heat=None, cfg=None)`.
3. Include metadata JSON with chain, token address, exit rate, max gain, tags.
4. Run: `python3 -m pytest tests/test_smart_money_scoring.py -q`.
5. Commit: `feat: score smart money signals`.

## Task 4: Add Cross-Source Resonance Engine

**Objective:** Merge market signals and smart money signals into stronger resonance signals and remove duplicates.

**Files:**
- Create: `src/banker_radar/signals/resonance.py`
- Test: `tests/test_resonance.py`

**Rules v0.3:**

- Normalize symbols:
  - `KAT-USDT-SWAP` -> `KATUSDT`
  - `KATUSDT` -> `KATUSDT`
- If a market signal and a smart signal share normalized symbol:
  - create `RadarSignal(kind="链上链下共振")`
  - score = max(market.score, smart.score) + configured bonuses, clamped to 100
  - risk = max risk of both, but high audit risk must not become positive resonance
  - reason = compact combination of both reasons
  - source = `market+binance-web3`
- Do not output the original two signals if resonance is emitted, unless `debug_keep_components` is configured.

**Steps:**
1. Write tests for symbol normalization.
2. Write tests for merge and duplicate suppression.
3. Implement `normalize_symbol()` and `merge_resonance(market_signals, smart_signals, cfg)`.
4. Run: `python3 -m pytest tests/test_resonance.py -q`.
5. Commit: `feat: add chain contract resonance engine`.

**Schema v0.3:**

```sql
CREATE TABLE IF NOT EXISTS radar_signals (
  id BIGSERIAL PRIMARY KEY,
  ts TIMESTAMPTZ NOT NULL,
  symbol TEXT NOT NULL,
  kind TEXT NOT NULL,
  score INTEGER NOT NULL,
  risk TEXT NOT NULL,
  reason TEXT NOT NULL,
  source TEXT NOT NULL,
  price_change_pct DOUBLE PRECISION NOT NULL DEFAULT 0,
  oi_change_pct DOUBLE PRECISION NOT NULL DEFAULT 0,
  funding_rate DOUBLE PRECISION NOT NULL DEFAULT 0,
  metadata_json JSONB NOT NULL DEFAULT '{}'::jsonb,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX IF NOT EXISTS idx_radar_signals_ts ON radar_signals (ts DESC);
CREATE INDEX IF NOT EXISTS idx_radar_signals_symbol_ts ON radar_signals (symbol, ts DESC);
CREATE INDEX IF NOT EXISTS idx_radar_signals_kind_ts ON radar_signals (kind, ts DESC);
CREATE INDEX IF NOT EXISTS idx_radar_signals_metadata_gin ON radar_signals USING GIN (metadata_json);
```

**Implementation guidance:**

- Use `psycopg` if available; otherwise add dependency in `pyproject.toml` as `psycopg[binary]`.
- Implement a small `create_store(config)` factory so CLI does not care whether backend is PostgreSQL or SQLite.
- Store `metadata_json` as JSONB in PostgreSQL and plain text in SQLite fallback.
- Provide an admin script or CLI helper later if needed, but for v0.3 keep schema creation inside store init.
- Never commit `DATABASE_URL`, PostgreSQL passwords, dumps, or `.env`.

**Steps:**
1. Write storage factory tests for `backend=postgres` and `backend=sqlite` selection.
2. Write PostgreSQL store tests that monkeypatch connection/cursor or run only when `DATABASE_URL` is present.
3. Implement `PostgresRadarStore.init()` with idempotent schema creation.
4. Implement `save_signals()` / latest-query parity with existing SQLite store.
5. Update CLI to call `create_store(config)` instead of directly constructing SQLite store.
6. Create local database/user using host PostgreSQL if missing, then put `DATABASE_URL` only in `/root/banker-radar/.env`.
7. Run: `python3 -m pytest tests/test_postgres_storage_v03.py tests/test_storage_factory_v03.py tests/test_storage_and_report.py -q`.

## Risk Notes

- Binance Web3 and GMGN output formats may differ from expectation. Keep parsers tolerant and test with captured examples.
- Some on-chain tokens do not map cleanly to futures symbols. v0.3 should skip unmatched tokens instead of guessing aggressively; GMGN symbol/address metadata should be kept in JSONB for future manual review.
- Too many Smart Money alerts may spam Telegram. Keep `smart_money.min_score` and section top-N strict at first.
- High exitRate and high audit risk must reduce/disable positive signals.
- Do not put Telegram bot token, Binance/GMGN credentials, private keys, PostgreSQL `DATABASE_URL`, or DB snapshots/dumps in git.
